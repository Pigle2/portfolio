FEATURE   PORTFOLIO / 2026

개발자로 7년, 기획자로 4년. 지금은 시스템의 근간을 아는 기획력으로 AI 팀을 만듭니다. 코드는 에이전트가 작성하고, 저는 무엇을 만들지 정합니다.

데이터와 시스템 구조에 대한 이해를 바탕으로, 다수의 AI가 24시간 일하는 시스템으로 비즈니스 목표를 달성하는 프로덕트를 설계합니다.

WHAT MAKES IT DIFFERENT

무엇이 다른가.

여러 에이전트가 서로 조율하고 검증하며 24시간 일하는 시스템을 설계합니다. 한 에이전트의 결과물을 다른 에이전트가 검증하고, 잘못된 곳은 다시 돌려 보냅니다. 사람이 자는 동안에도 시스템은 스스로 진행합니다.

HOW I WORK

작업 방식.

AI로 프로젝트를 진행할 때 사용하는 8가지 디테일.

01

Claude Code × Codex 크로스 확인

한 모델이 작성한 산출물을 다른 모델이 다시 검증합니다. 단일 모델에 의존할 때 발생하는 오판과 누락을 시스템 차원에서 걸러냅니다.

02

9단계 SDLC

자료조사 → 플로우 → 기능명세서 → 상세기획서 → 아키텍처 설계 → 구현 → 산출물 검증 → E2E 테스트 → 보안 점검. 각 단계는 독립된 게이트를 가지며, 사용자 확인 없이 다음 단계로 넘어가지 않습니다.

03

하네스 개념

에이전트가 자유롭게 판단하되, 판단의 결과는 사전에 정의된 하네스 (통제 환경) 안에서만 작동합니다. 품질과 자율성의 경계를 문서로 고정합니다.

04

에이전트 팀

위 9단계를 각각 다른 에이전트가 담당합니다. 기획, 설계, 구현, 검증, 테스트, 보안 에이전트가 각자의 역할을 수행하고 오케스트레이터가 조율합니다. 필요에 따라 새 에이전트를 동적으로 생성합니다.

05

자기개선 루프

시스템이 주기적으로 자기 프로세스의 병목과 실패 패턴을 찾고 작업 규칙을 스스로 갱신합니다. 사람의 개입 없이 다음 사이클에 반영됩니다.

06

워크트리 격리

모든 에이전트는 격리된 git 워크트리에서 작업합니다. 멀티 에이전트가 동시에 같은 저장소를 편집할 때 발생하는 머지 충돌을 구조적으로 예방합니다.

07

24시간 스케줄링

스케줄러가 잠들지 않습니다. 사용자가 자고 있어도 에이전트는 KM 프로젝트의 다음 단계를 자율적으로 진행합니다. 아침에 확인하는 것은 완료된 산출물입니다.

08

km/CLAUDE.md — 작업 규칙 문서

위 7개의 작업 원칙과 에이전트 규칙이 KM 프로젝트의 CLAUDE.md 파일에 모두 명문화되어 있습니다. 말로 하는 방식이 아니라 문서가 강제하는 방식입니다. 아래 KM 섹션에서 발췌로 확인할 수 있습니다.

CASE STUDY

KM.

부동산 경매 물건 탐색 시스템.

→ 라이브 보기

KM은 수많은 부동산 경매 매물 중에서 좋은 물건을 찾아내는 프로그램입니다. 현재 개발 중이며, 위에서 설명한 8가지 작업 체계가 그대로 적용되는 첫 본격 프로젝트입니다.

역할 분담은 명확합니다. Claude가 오케스트레이터로서 기획·설계·개발을 주도하고, 사용자는 비용이 발생하는 결정과 단계 전환 승인만 합니다. 에이전트는 자율적으로 판단하고 실행하되, 모든 산출물은 크로스 리뷰를 거쳐 승인됩니다.

아래는 km/CLAUDE.md에 명문화된 세 가지 작업 원칙입니다.

km/CLAUDE.md — Cross-Review 원칙
모든 에이전트 산출물은 Claude(오케스트레이터)가 반드시 검토 후 승인한다.
에이전트가 만든 결과물을 그대로 적용하지 않는다 — 항상 크로스 체크.
품질 미달 시 피드백과 함께 에이전트에게 재작업을 지시한다.
에이전트 간 충돌 시 Claude가 최종 결정한다.
km/CLAUDE.md — Agent System
Claude는 직접 작업하지 않고, 에이전트에게 지시하여 작업을 수행한다.
에이전트는 백그라운드로 실행 — 대화가 끊기지 않도록.
겹치지 않는 작업은 병렬로 진행 — 독립 작업은 동시에 여러 에이전트 투입.
여러 에이전트가 같은 파일을 동시에 수정하지 않는다 — 한 파일은 한 에이전트만.
km/CLAUDE.md — 스케줄 & 워크트리
스케줄 등록/변경/삭제 시 /docs/schedules.md에 반드시 반영한다.
새 세션 시작 시 schedules.md를 읽고 활성 스케줄을 재등록한다.

에이전트는 feature 브랜치(워크트리)에서 작업한다.
산출물 유형별 병합 조건 충족 확인 후 main에 병합.

KM은 현재 구현 중입니다.
