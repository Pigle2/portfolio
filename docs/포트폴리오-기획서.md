## 1. 프로젝트 목적
임상준의 새 포트폴리오. **AI를 활용해 프로젝트를 진행/관리하는 직무로의 커리어 전환을 어필**하는 것이 핵심 목적. 기존 7년 백엔드 + 3년+ 기획 경력은 "AI 활용 판단력의 신뢰 근거" 보조 컨텍스트로만 사용.

이전 v1.0.0(dark editorial)이 경력/PM 어필 중심이었다면, **v3는 "본격적인 AI 운영 시스템을 들고 있는 기획자"** 어필로 콘셉트 이동.

---

## 2. 타깃 독자
**AI 활용 인재를 뽑으려는 회사의 실무자 / 관련팀** (HR/채용담당자 아님).

한국 채용 프로세스상 실무자가 검토하는 게 일반적. 따라서 1~2분 훑기 시나리오는 부적합 — **실무자가 "이 사람 우리 팀에 데려와서 바로 일 시킬 수 있겠다"라고 느낄 만한 구체 사례·툴·결과물·디테일이 필요.**

## 3. 메시지 코어

### 3.1 한 줄 정체성
**"관점이 다른 기획자"** — 개발자는 아니지만 AI로 직접 만들 수 있는 사람. 기획 강점이 있고, **결과물로 신뢰 확보**한다. "신뢰"가 핵심 어휘.

### 3.2 사용자 원문 (인터뷰 Q2)
> "현재 개발자가 아니지만 ai로 개발을 할 수 있다는것이야..관점이 다르다는거지..기획에 대한 강점이 있는거지..이 내용을 어필해야하고 그리고 개발한것을 보여줘서 믿을수 있게 해야해 중요해"

### 3.3 진짜 차별점 — AI 운영 시스템 (8개 디테일)
이 8개는 모두 사용자가 직접 언급한 사실. **사용자가 명시적으로 "포트폴리오에 빠짐없이 표현해"라고 지시한 항목들. 빌드 시 누락 금지 체크리스트.**

1. **Claude Code + Codex 조합 → 크로스 확인 (cross-check)** — 한 모델이 한 일을 다른 모델이 검증
2. **9단계 SDLC (내부 운영 단계명 기준)** — 자료조사 → 플로우 → 기능명세서 → 상세기획서 → 아키텍처 설계 → 구현 → 크로스 리뷰 → E2E 테스트 → 보안 점검  
   *사이트 외부 노출 시 "코드 리뷰" 표현 금지(5.1 포지셔닝 제약). 대외 문구는 "산출물 검증" 또는 "크로스 리뷰"로.*
3. **하네스(harness) 개념** 사용
4. **에이전트 팀** — 위 9단계를 각각 다른 에이전트가 담당, 상호 보완
5. **자기개선 루프** — 주기적으로 스스로 문제 찾아 개선. **2026-04-08 구현 완료.** 이전 "미구현 → 구현했다로 표현" 결정은 실구현으로 해소됨. 진정성 룰(5.7)과의 충돌 제거.
6. **워크트리(git worktree) 활성화** — 에이전트들이 격리된 워킹 디렉토리에서 작업 → git 충돌 예방. 멀티 에이전트 동시 작업의 핵심 기술 함정 해결.
7. **스케줄링 → 24시간 개발** — 사용자가 자고 있어도 스케줄된 에이전트가 km 프로젝트를 계속 진행. 비개발자 출신 기획자가 이 운영 체계를 직접 설계하고 운영함.
8. **km/CLAUDE.md** — 위 시스템 전체가 명문화된 운영 규칙 문서. 포트폴리오에서 발췌해서 직접 보여줄 핵심 자산.

## 5. 포지셔닝 제약조건 (v1.0.0에서 살린 룰 — v3에도 적용)
콘텐츠 작성 시 반드시 준수. 위반 시 메시지 코어와 어긋남.

### 5.1 금지 표현
- "코드를 읽고", "API를 설계하는", "코드 레벨 분석", "코드 리뷰"
- "Technical PM"
- "프롬프트 엔지니어링" (할 줄 모름)
- "코드로 기획"
- 100만 DL / 1000억 거래 등 과거 백엔드 시절 성과를 **대표 문구·타이틀·태그라인**으로 사용 금지 (본문 내 자연스러운 언급은 허용)

### 5.2 올바른 표현
- "개발 경험을 바탕으로"
- "7년간의 서버 개발 경험으로"
- "개발을 이해하는 기획자"

### 5.3 원칙
- 임상준은 현재 코드를 읽지 않고, API를 설계하지 않음. 코드를 보지 않고, 보고 싶지도 않음.
- 100만 DL은 백엔드 개발자 시절의 성과이지, PM으로서의 성과가 아님
- 대표 문구는 "뭘 할 수 있는지"(역량) 중심. "뭘 했느냐"(과거 성과) 중심 금지
- 현재 역할은 "기획자"이지 "Technical PM"이 아님
- 현재 역할을 과장하지 말 것. 과거 경험은 정확한 역할로 귀속할 것

### 5.4 사실 관계
- **현대차 그룹사 시작일:** 2025.10
- **루덴스플레이 / 플라이북:** 별개의 회사. 합쳐서 표기하지 말 것

- **노출 순서가 핵심.** 사용자가 명시한 강조점 — 사이트 구조 = 콘텐츠가 어떤 순서로 등장하는가 = 메시지의 흐름.

### 5.7 카피·문구 톤 룰 (사용자 명시)
- ✅ **진정성 우선.** 모든 카피/문구는 있는 그대로의 사실을 가장 짧고 정확하게 압축한 것이어야 함. 사실 아니면 폐기. 본문/결과물이 증명 못 하면 폐기. 과장·꾸밈·연출 금지.
- ❌ **카피라이팅 잡기술 금지.** "임팩트", "호기심 자극", "도발", "후킹", "예상 깨기" 같은 평가 프레임으로 카피 만들지 말 것. 사용자 원문: "임팩트나 호기심 이딴 잡기술 말고 그냥 진정성이 중요하잔아."
- ❌ **비격식 어휘 금지.** "굴리다 / 좀 / 그냥 / 막 / 이딴" 같은 구어/비격식 표현은 본문 카피에서 사용 금지. 사용자가 직접 "굴리다 → 운영하다"로 교정한 사례 참조. 격식 있는 표현 사용.

## 6. 사이트 구조 — 한 페이지 긴 에디토리얼 스크롤

## 7. 디자인 — v2/ 매거진 시스템 활용
**시각언어:** v2/index.html, v2/style.css의 디자인 시스템을 v3 콘텐츠에 입힘. 상세:

### 7.1 폰트
- **Playfair Display** — 영문 디스플레이/타이틀
- **Cormorant Garamond** — 세리프 본문, 인용
- **Noto Sans KR** — 한글 본문/타이틀

### 7.2 레이아웃 / 시각 요소
- 비대칭 grid, 우아한 여백, 구분선(divider)으로 섹션 분리
- 색감: v2/ dark editorial 그대로 (정확한 색상 값은 v2/style.css 참조)
- 매거진 라벨 ("FEATURE / 2026" 같은 메타 헤더 텍스트)
- 인용/발췌(km/CLAUDE.md 같은 문서 발췌)는 monospace 또는 인용 박스 + 큰 따옴표 처리

### 7.3 금지 (5.6 재확인)
- 카드 그리드
- 균일한 박스 나열
- AI가 성의없이 만든 듯한 일반 포트폴리오 패턴

### 7.4 빌드 방식
v2/style.css를 base로 가져와서 v3 콘텐츠 구조에 맞게 일부 컴포넌트만 추가/조정. 폰트 import, 색상 변수, 타이포 스케일은 v2 그대로 재사용. v2/, v1/ 폴더는 절대 건드리지 말 것.
