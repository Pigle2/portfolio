## 1. 내적 일관성
- 발견: `Outputs`는 닫혔다고 했는데, km 노출 형식의 핵심 디테일은 아직 열려 있다.
  근거: `[결정 대기 항목 L28-29] 'Outputs ... 모두 닫힘'` / `[4.1 L96] 'km/CLAUDE.md 발췌 + 내러티브 글 + 라이브 링크(추후) — 디테일은 결정 대기.'`
  영향: 빌드 전에 확정돼야 할 노출 형식이 미정이라 `Outputs 닫힘` 판단이 성립하지 않는다.
  제안: `Outputs`를 다시 열거나, 최소한 `발췌 범위/라이브 링크 유무/발췌 형식` 3가지를 본문에 확정값으로 적어 닫아야 한다.

- 발견: `사실만 쓰기` 원칙과 `미구현 항목을 구현했다고 표기` 결정이 정면 충돌한다.
  근거: `[3.3 L59] '이 8개는 모두 ... 사실'` / `[3.3 L65-66] '현재 미구현 ... 포트폴리오에는 "구현했다"로 표현'` / `[5.7 L177] '사실 아니면 폐기'`
  영향: v3의 핵심 자산인 운영 시스템 신뢰가 가장 중앙에서 무너진다.
  제안: 자기개선 루프는 `구현 중` 또는 `운영 규칙으로 설계 완료`처럼 사실 상태로 낮추거나, 실제 구현 완료를 빌드 게이트로 올려야 한다.

- 발견: 금지 표현으로 묶어둔 `코드 리뷰`가 8개 핵심 디테일 안에 그대로 포함돼 있다.
  근거: `[5.1 L145] '"코드 리뷰" 금지'` / `[3.3 L62] '9단계 SDLC ... 코드 리뷰 ...'`

  제안: `코드 리뷰`를 사이트 카피에서는 `산출물 검증` 또는 `크로스 리뷰`로 치환하고, 내부 운영 단계명과 대외 표현을 분리해야 한다.

- 발견: 9단계 SDLC와 km의 4단계 흐름 관계가 설명 없이 병치돼 해석이 갈린다.
  근거: `[3.3 L62] '9단계 SDLC'` / `[4.1 L84] '4단계 흐름: 기획 → 설계 → 구현 → 테스트'`

## 6. 개선 제안

**강화:**
- `How I Operate` 안에 8개 각각을 "한 줄 정의 + 한 줄 증거" 구조로 고정. 특히 하네스와 자기개선 약함.
- "운영 헌법"을 "문서가 있다"보다 "문서가 어떻게 행동을 강제하는가"로 보강. Cross-Review, Planning-First, 스케줄 관리 각 1개씩 실제 적용 사례.
- v2 매거진 언어를 빌리되 "헌법 발췌"는 시각 장식이 아니라 증빙 블록으로. v2의 가져올 요소(`hero-badge`, 폰트/세리프 조합)와 버릴 요소(`stats-grid`, `project-card`) 분리 명시.

**축소:**
- 반응 유도형 문장(`[3.4 L72]`, `[6.2 L205]`) 제거.
- "다른 사람들이 'AI 잘 써요'" 같은 비교 문장은 사실 증명보다 상대 폄하로 읽힐 수 있음.
- v1과의 차이/콘셉트 이동/변경 이력 메타 설명 반복은 사이트 본문에서 최소화.

**명확화:**
- 9단계와 4단계의 관계를 "상위 페이즈/하위 운영 단계"로 한 번에 정의.
- "기발자"는 사용자 표현이라도 "현재 역할은 기획자"와 함께 읽히도록 보정 문장 필요.
- 결정 대기 "닫힘/열림" 상태 표기 다시 정합.
